## Worked example: force-shutdown gap (F1)

At baseline, quitting the deck killed every agent ‚Äî period. `src/main.rs:413‚Äì418` in the baseline:

```rust
// TUI exited ‚Äî clean up
daemon_handle.abort();

if path.exists() {
    let _ = std::fs::remove_file(&path);
}
```

The daemon was an in-process tokio task (`tokio::spawn` at baseline `src/main.rs:378`); the agent PTYs were `tokio` children of the deck process (`pane::detect_multiplexer` returning an in-process `PaneController`). Exiting the TUI aborted the daemon task, removed the hook socket file, and let the deck process exit ‚Äî taking every agent PTY with it. The baseline quit dialog (`docs/keyboard-shortcuts.md:93`) was a literal **Yes / No** confirmation that quitting would happen.

In current code (post PRD #76 + PRD #93):

- The daemon is fork-execed detached via `setsid(2)` into its own session (`src/daemon_attach.rs:343`); the comment at `src/main.rs:626‚Äì630` makes this explicit ‚Äî *"the daemon was fork-execed detached by `ensure_external_daemon_or_die` (setsid'd into its own session) so it is intentionally outside this process tree: we do not abort the daemon and do not unlink its sockets. Agents must survive TUI exit (PRD #76 line 199)."*
- Agent PTYs are owned by the daemon's `AgentPtyRegistry` (`src/agent_pty.rs:805+`), spawned via `daemon.rs:709‚Äì710`, well outside the TUI's process tree.
- The quit dialog was deliberately collapsed to **Detach / Cancel** per PRD #93 M4.2 ‚Äî *"every pane is daemon-backed so quitting the TUI is always a detach, never a kill"* (`src/ui.rs:1666‚Äì1700, 535‚Äì536, 599, 5328‚Äì5339`).
- The daemon idle-shuts-down only when *both* `clients == 0 AND agents == 0` (`src/daemon.rs:480‚Äì551`), so as long as any agent is alive, the daemon stays up ‚Äî by design (PRD #93 line 32).
- **Neither `DaemonCmd::Stop` nor `RemoteCmd::Stop` exists.** `DaemonCmd` has only `Serve` and `Hello` (`src/main.rs:135‚Äì150`); `RemoteCmd` has only `Add`, `List`, `Remove`, `Upgrade` (`src/main.rs:160‚Äì209`); `Remove` deregisters a local registry entry, it does not touch the remote daemon.

PRD #93 line 39 anticipated this exact gap: *"User can `dot-agent-deck remote stop` (or equivalent local command) to force shutdown."* The promise was never met. Today's only way to stop the daemon while agents are managed is `pkill dot-agent-deck`. This is the parity regression at the heart of the audit ‚Äî the same user gesture (quit the deck) used to kill the agents, and now no in-product gesture does.

This is the only Regressed row in the audit. Everything else is Preserved or Intentional change with citation.

### F1 ‚Äî Add a force-shutdown gesture (shipped as Ctrl+C dialog "Stop")

**Problem**: At baseline, quitting the deck killed every agent. After the PRD #76 + PRD #93 pivots, agents are owned by a setsid'd daemon process that intentionally outlives the TUI, idle-shutdown is gated on `agents == 0`, and the quit dialog was deliberately collapsed to Detach/Cancel ‚Äî leaving no in-product gesture that stops a running deck installation. PRD #93 line 39 anticipated needing a force-shutdown command but neither `DaemonCmd::Stop` nor `RemoteCmd::Stop` shipped.

**Status**: **Shipped** via PRD #92 Phase 6 (M6.1 design lock / M6.2 implementation / M6.3 tests). Implementation:

- Primary Ctrl+C dialog now has three options ‚Äî Detach (0, default) / Stop (1) / Cancel (2) ‚Äî in `src/ui.rs::handle_quit_confirm_key` and rendered by `src/ui.rs::render_quit_confirm`. Detach stays the default so the existing muscle memory does not become destructive.
- Picking Stop with 0 managed agents proceeds directly to `KeyResult::StopAndQuit`. With ‚â•1 agents, the TUI transitions to a secondary `UiMode::StopConfirm` y/n dialog (`src/ui.rs::handle_stop_confirm_key`, `src/ui.rs::render_stop_confirm`) that defaults to **No** and names the agent count in the prompt.
- `KeyResult::StopAndQuit` calls `EmbeddedPaneController::shutdown_daemon` (`src/embedded_pane.rs::shutdown_daemon`), which sends a new `KIND_SHUTDOWN` (`0x15`, `src/daemon_protocol.rs:113`) frame via `DaemonClient::send_shutdown` (`src/daemon_client.rs::send_shutdown`).
- Daemon-side: `handle_connection` in `src/daemon_protocol.rs` short-circuits `KIND_SHUTDOWN` before the usual `KIND_REQ` decoding, calls `AgentPtyRegistry::shutdown_all_graceful(Duration::from_secs(3))` (`src/agent_pty.rs::shutdown_all_graceful` ‚Äî SIGTERM each child, poll `try_wait` up to 3s, then SIGKILL survivors via the existing teardown), and signals the daemon's shutdown `Notify`. The hook loop exits, `run_daemon_with` returns, and the registry's Drop calls `shutdown_all` as the backstop.
- Idempotency: `AgentPtyRegistry::shutting_down: AtomicBool` latches on first entry so a second `KIND_SHUTDOWN` (or a SIGTERM-triggered drop racing the protocol handler) is a no-op.

### F4 ‚Äî Ctrl+W respects `close_pane` errors

**Problem**: Auditor-found post-PRD-#76 regression. `EmbeddedPaneController::close_pane` is explicitly designed to return `Err` when the daemon's `StopAgent` RPC fails or times out (the controller restores the local pane state internally so the user can retry ‚Äî `tests/daemon_attach_cleanup.rs::ctrl_w_stop_agent_timeout_restores_pane_and_returns_error` pins this contract). The TUI ignored the result: `src/ui.rs:3736‚Äì3780` did `let _ = pane.close_pane(pane_id)` and then unconditionally removed the dashboard session, card, and pane metadata. Outcome: a failed StopAgent left the agent alive in the daemon registry while the card vanished from the dashboard ‚Äî the user had no way to see the agent and no way to retry the close. The group-close paths (`TabManager::close_tab`, `ModeManager::deactivate_mode`) had the same shape but worse: a single failed pane silently destroyed the whole tab.

**Status**: **Shipped** (PRD #92 Phase 8 ‚Äî M8.1 / M8.2 / M8.3).

- **M8.1 single-pane (`src/ui.rs`)**: The Ctrl+W arm now matches on `pane.close_pane(pane_id)`. On `Ok(())` the session, card, and pane metadata are removed as before. On `Err(e)` they stay, a `tracing::warn!` records the error, and `ui.status_message` is set to a retry-hinting message (`"Failed to close pane {id}: {e} ‚Äî press Ctrl+W to retry"`).
- **M8.2 group close**: New [`CloseTabOutcome`](../src/pane.rs) type in `src/pane.rs` with `closed: Vec<String>` and `failed: Vec<(String, String)>`. `TabManager::close_tab` (`src/tab.rs`) and `ModeManager::deactivate_mode` (`src/mode_manager.rs`) now return per-pane outcomes; the Ctrl+W dispatcher builds a `HashSet<&str>` from `outcome.closed` and uses it to filter session removal, so failed panes keep their dashboard cards. `tracing::warn!` per failure plus an aggregate status-bar message ("Close partially failed for N pane(s) ‚Äî first error: ...").
- **M8.3 tests (`tests/close_pane_errors.rs`)**: 4 new integration tests cover the happy path (no failures), orchestration group-close with one failing pane (split between `closed` and `failed`), mode-tab teardown where the agent pane close fails (side panes still close, agent retains its card), and `ModeManager::deactivate_mode` directly with one failing persistent pane. All 4 pass.

### F5 ‚Äî process-group kill semantics

**Problem**: Auditor-found defect (possibly pre-existing ‚Äî baseline's `pane.child.kill()` had the same single-PID limitation, but the user-visible symptom is now sharper because the daemon outlives the TUI). Commands launched via `$SHELL -c <cmd>` register the shell's PID, not the actual agent's. `force_kill_child_and_wait` sent `SIGKILL` to that recorded PID only ‚Äî direct child only, no process-group semantics. The shell died; the shell's children (the actual agent, plus anything the agent spawned ‚Äî language servers, file watchers, etc.) were orphaned and re-parented to PID 1. Daemon and TUI both thought the kill succeeded. `tests/local_attach.rs:257‚Äì299` only asserted the recorded child PID dies, not its descendants, so the test suite missed this.

**Status**: **Shipped** (PRD #92 Phase 9 ‚Äî M9.1 / M9.2 / M9.3).

- **M9.1 spawn-side process-group setup**: no code change required. `portable-pty` already calls `setsid()` in its `pre_exec` on Unix (`portable-pty-0.8.1/src/unix.rs:220`), so every PTY-spawned child is a session leader by construction ‚Äî its PID equals its session ID and its process-group ID. The original F5 plan anticipated adding a `setpgid(0, 0)` `pre_exec` hook to the spawn path in `src/agent_pty.rs`, but that step would have been redundant.
- **M9.2 kill-side update**: `force_kill_child_and_wait` (used for single-pane Ctrl+W via StopAgent, and indirectly by `shutdown_all` from Drop) now calls `libc::killpg(pid as i32, SIGKILL)` instead of `libc::kill(pid as i32, SIGKILL)`. The SIGTERM phase inside `shutdown_all_graceful` (used by the F1 KIND_SHUTDOWN path) likewise calls `libc::killpg(pid, SIGTERM)`. Both signal the entire process group rather than just the direct child. Used `libc::killpg` directly rather than pulling `nix` in for one wrapper.
- **M9.3 tests (`tests/process_group_kill.rs`)**: `close_pane_reaps_shell_descendants` launches `sh -c 'sleep 30 & echo $! > FILE; wait $!'` so the shell writes its background `sleep`'s PID to a relay file. The test reads that PID, asserts both the shell and the descendant are alive (so a descendant that died on its own can't make the test pass for the wrong reason), calls `close_pane`, and probes both PIDs with `libc::kill(pid, 0)` returning ESRCH within 3s. The 3s bound is well under the descendant's 30s sleep, so a pre-F5 regression would leave the sleep alive past the bound and fail the assertion. 1 test, passes.

**Limitation (surfaced during F5 manual-test pass, 2026-05-23)**: `killpg` reaches the agent's session but does NOT cross `setsid` boundaries that the agent itself creates. Claude Code internally `setsid`s its sub-shells, placing them in fresh process groups that escape the outer `killpg`. Those orphaned sub-shells survive Ctrl+W as init-parented processes. Mitigation: **F8** (graceful single-pane close ‚Äî SIGTERM with 3 s grace before SIGKILL) gives the agent a catchable signal so it can reap its own descendants during the grace window. See F8 below for details. If the agent ignores SIGTERM or doesn't clean up, the descendants survive ‚Äî that's an agent bug, not a deck bug.

### F8 ‚Äî Graceful single-pane close (SIGTERM-then-SIGKILL)

**Problem**: F5's `killpg(pgid, SIGKILL)` kills the agent's immediate process group but cannot reach sub-sessions the agent itself creates via `setsid` (real-world example: Claude Code's internal sub-shells). The pre-F8 Ctrl+W path also sent SIGKILL directly, which is uncatchable ‚Äî so even a well-behaved agent that *wanted* to clean up its own descendants had no opportunity to do so.

**Status**: **Shipped** (PRD #92 Phase 10 ‚Äî M10.1 / M10.2).

- **M10.1**: new `terminate_child_with_grace_and_wait` helper in `src/agent_pty.rs` ‚Äî SIGTERM the child's process group, poll `try_wait` for up to `AGENT_TERMINATE_GRACE` (3 s), then SIGKILL as the backstop. `AgentPtyRegistry::close_agent` (the single-pane Ctrl+W path) now routes through this helper. `force_kill_child_and_wait` is retained as SIGKILL-only for the contexts where a grace window is wrong or unnecessary (`PtyGuard` / `AgentPty` `Drop`, `shutdown_all` from registry `Drop`, `shutdown_all_graceful`'s phase-3 SIGKILL backstop). Both paths now share a new `signal_child_pgroup_or_fallback` low-level helper for the `killpg` syscall + `pid_to_pgid` validation + `tracing::warn!` shape, so the three call sites (single-pane SIGTERM, single-pane SIGKILL fallback, daemon-wide SIGTERM) can't drift.
- Controller-side timeout adjustment: the Ctrl+W path's `CREATE_PANE_STOP_TIMEOUT` (2 s ‚Äî written for the attach-failure cleanup) was too tight after F8 because the daemon-side RPC can now legitimately take up to 3 s. New dedicated `CTRL_W_STOP_TIMEOUT = 5 s` (3 s F8 grace + 2 s buffer) for the Ctrl+W path; the 2 s constant stays in place for its original use.
- **M10.2 tests (`tests/process_group_kill.rs`)**: `close_pane_well_behaved_agent_runs_sigterm_trap` (shell traps SIGTERM, writes a sentinel, exits; test polls for the sentinel within 2 s of the close) and `close_pane_uncooperative_agent_killed_after_grace` (shell ignores SIGTERM via `trap '' TERM`; test asserts the close completes under 3.5 s ‚Äî well above the 3 s grace, well below the shell's 60 s `sleep`). Existing `daemon_attach_cleanup::ctrl_w_stop_agent_timeout_restores_pane_and_returns_error` adjusted: outer deadline 4 s ‚Üí 7 s to match the new `CTRL_W_STOP_TIMEOUT`.

The deck's contract: "the agent gets a catchable signal and a bounded grace window." The agent's contract: "use that signal to reap your own descendants if you spawned any." Agents that don't honour the contract still leak descendants ‚Äî visible via `pgrep` after Ctrl+W ‚Äî but the deck has no way to chase arbitrary descendant trees safely without crossing `setsid` boundaries that exist for good reasons.

### F9 ‚Äî Restore worker context cleanup per `.dot-agent-deck.toml` `clear` setting

**Status**: Shipped (respawn-based). Implementation matches baseline `2fc39c3:src/ui.rs::dispatch_delegate_events` semantics: on a `clear = true` delegate the daemon terminates the worker's existing child (SIGTERM with the F8 3 s grace then SIGKILL) and spawns a fresh agent reusing the same `pane_id_env` before writing the new prompt. `clear = false` skips the respawn. Skip the clear when `clear == false` (release role keeps its scrollback). Tests: parse-level for default and explicit values; dispatch-level for honoring of the setting.

**Likely PRD home**: PRD #92 Phase 11 (this audit's remediation scope).

### F12 ‚Äî TUI auto-renew per-pane subscription on respawn

**Status**: Shipped (user manual-tested 2026-05-24). Final fix at commit `96762b3`; e2e regression test at commit `a2b5370`. Folded into scope 2026-05-24 after manual testing of F9 followup-7 confirmed the daemon side was fully working but the TUI side did not auto-renew its `broadcast::Receiver` subscription when an agent was respawned.

*F12 saga (commits in order):* `6368a8f` (initial reattach loop ‚Äî passed unit tests, failed manual test) ‚Üí `7c6356f` (close_pane retry-once on stale agent_id + doc nits) ‚Üí `96762b3` (the REAL fix: replaced the 200 ms retry budget with exponential backoff totaling ~10 s; the bug was that the production respawn gap of 0.5‚Äì3 s exceeded the 200 ms budget) ‚Üí `a2b5370` (e2e regression test that exercises the real `respawn_agent_for_pane` path with a SIGTERM-trapping shell, FAILS at the broken budget, PASSES at the fixed budget) ‚Üí `d14f7bc` (doc-comment + test-name polish).

*Fix lives in `src/embedded_pane.rs::run_pane_io_task`* ‚Äî the per-pane io_task wraps its read/write `select!` in an outer reconnect loop. On STREAM_END from the daemon, the loop calls `resolve_and_reattach` to look up the NEW agent for `pane_id_env` via `list_agents` (with **exponential backoff: 200 ms initial doubling to 1 s cap, ~10 s total budget**) and re-`attach`. `StreamBackend::agent_id` is now an `Arc<Mutex<String>>` shared with the io_task and the resize worker so `close_pane` / `resize_pane_pty` target the currently-bound agent.

*Tests:* `tests/pane_auto_renew_on_respawn.rs` covers four unit-level scenarios (respawn happy path, no-live-agent give-up, respawn-in-flight race, no-respawn regression) but bypasses the real daemon dispatch ‚Äî those tests passed at the broken budget. **Problem**: When the daemon respawns an agent (F9 `clear = true` path), the TUI's per-pane stream subscriber is bound to the OLD agent's `broadcast::Sender`. The OLD agent dies, the sender is dropped, the receiver returns `Closed`. The TUI does not re-resolve the pane to the NEW agent's sender. The pane view freezes showing only stale output (typically just the devbox bootstrap line); the user must manually detach and re-attach to see the new agent's output. Verified empirically by the user: detach + re-attach immediately reveals the running new agent. The F9 coder's claim that *"the TUI's existing reconnect machinery handles this"* is incorrect ‚Äî that machinery handles whole-daemon reconnect, not per-pane agent-respawn.

**Suggested approach**: TUI-side fix. On `broadcast::error::RecvError::Closed`, the per-pane stream subscriber re-resolves `pane_id_env` via the attach protocol's `list_agents`, finds the NEW agent's id, and re-subscribes. Small backoff (e.g. 100 ms initial) for the case where Closed fires before the new agent has registered. Cap total retries so a permanently-closed pane doesn't spin.

**Likely PRD home**: PRD #92 Phase 13 (this audit's remediation scope).

## Intentional changes appendix

Behaviors that changed between baseline and current, where the change is a deliberate design decision with citation. Recording so a future re-audit does not re-flag.

### ¬ß A. Quit confirmation dialog: Yes/No ‚Üí Detach/Cancel

**Where**: `src/ui.rs:1666‚Äì1700, 5328‚Äì5339`.

**Justification**: PRD #93 M4.2 explicitly collapses the dialog. Every pane is now daemon-backed, so the TUI cannot kill agents from inside the dialog regardless of which option the user picks; offering a "Yes (quit)" that no longer kills anything would be a misleading UX. The fact that the architecturally-correct dialog now reads "Detach / Cancel" is the visible symptom of F1 ‚Äî the user can no longer reach the kill path that "Yes" used to perform.

### ¬ß B. `DOT_AGENT_DECK_SOCKET` `/tmp` fallback gains uid suffix

**Where**: `src/config.rs:52‚Äì68` (`socket_path`) and mirror at `:76‚Äì91` (`attach_socket_path`).

**Justification**: Introduced in commit `48b9180` (the PRD #93 squash-merge that brought in always-external-daemon work). The inline code comment at `src/config.rs:61‚Äì67` records the rationale: *"the `/tmp` fallback must include the uid so two users on the same host can't collide on the same socket path (the daemon is per-user; the 0o600 mode is on the socket inode, but the *path* still has to be unique, otherwise the loser's `bind(2)` sees `EADDRINUSE` against the winner's inode). Same rationale as `attach_socket_path` below."* The PRD document itself does not contain a numbered REV-2 reference; the citation lives in the squash-merge commit message and the code comment. On a shared-host where `$XDG_RUNTIME_DIR` is unset and multiple users fall back to `/tmp`, the original `/tmp/dot-agent-deck.sock` would be a collision target. The new fallback is `/tmp/dot-agent-deck-{uid}.sock`. The env-var override behavior is unchanged; only the `/tmp` default differs. ### ¬ß C. In-process daemon ‚Üí external daemon
