# Config-gen baseline regeneration (PRD #116)

> **Developer / maintainer reference.** This page documents the repeatable analysis workflow behind PRD #116 — regenerating what the config generator *would* produce for a project and diffing it against the user's hand-improved `.dot-agent-deck.toml` to drive prompt/role-library edits. It is intentionally excluded from the published documentation site and renders as plain Markdown here on GitHub.

The deck ships an AI config generator: an agent reads a project, follows `assets/config_gen_prompt.md` (plus the role library `assets/roles.toml`), and writes an initial `.dot-agent-deck.toml`. Users then hand-edit that file. PRD #116 treats the set of hand-improved configs as a training set: reconstruct the AI baseline, diff it against the user config by region, aggregate recurring deltas into a catalogue, push targeted edits back into the prompt/roles, and re-run to confirm the new baseline is closer. This doc is how you re-run it as more projects accumulate or as the prompt evolves.

## Where the tooling lives

- `examples/render_config_gen_prompt.rs` — prints the **exact** post-render prompt the deck would send for a directory (`config_gen::config_gen_prompt`, with `{dir}` and `{roles}` interpolated). This is the auditable "what was sent" capture.
- `examples/diff_config.rs` — region-structured diff between a regenerated baseline and a user config, parsed with the deck's own `ProjectConfig` types so field semantics (defaults for `watch`, `reactive_panes`, `clear`, …) match the running binary. Emits Markdown grouped by region.
- `assets/config_gen_prompt.md`, `assets/roles.toml` — the prompt and role library being studied/edited. They are embedded into the binary via `include_str!` in `src/config_gen.rs`, so editing the asset is enough to change behavior (rebuild required for the example to pick it up).
- `analysis/prd-116/` — all outputs (per-project artifacts, the pattern catalogue, the validation write-up). This directory lives outside `docs/`, so it is not published; do **not** add it to `site/sidebars.js`.

Build the toolchain with the project's pinned Rust (rustup shims are broken in this repo — see the toolchain note in the team memory): prefix `PATH` with `.devbox/nix/profile/default/bin` or run inside `devbox shell`.

## The engine (pin this exactly, or baselines are not comparable)

Baselines are generated by the **authenticated `claude` CLI in print mode**, single-shot, with all tools disabled and the project laid out inline. This is the reproducible stand-in for the deck's real config-gen flow (the rendered prompt is fed to a `claude`/`opencode` agent pane via `send_config_gen_prompt`); the deck does **not** use the `src/llm.rs` API client for config-gen.

> **Why the CLI and not the raw API — a recorded, user-signed-off decision.** PRD #116's original decision #1 proposed sending the rendered prompt to the **Anthropic API** with **temperature 0** for determinism. The engine below — the `claude` CLI single-shot — is a *deliberate, user-confirmed* deviation from that wording, approved before the pilot. It is the right call because it is **more faithful to how the deck actually generates configs**: the deck feeds the rendered prompt to a live `claude`/`opencode` **agent pane** (`send_config_gen_prompt`), not to the `src/llm.rs` API client. The accepted trade-off is that **the `claude` CLI exposes no temperature flag, so temperature 0 cannot be pinned**; determinism was therefore checked **empirically over repeated runs** with the model pinned to `claude-haiku-4-5`, and reinforced by preferring cross-project recurrence over single-project re-sampling (below). Treat this as the standing engine of record, not a fallback.

- **Model — pinned:** `claude-haiku-4-5` (the deck's documented default, `src/config.rs`). Do not substitute Sonnet/Opus: a stronger model would understate the baseline gap.
- **Tools — all disabled.** The project is already discovered for you and pasted into the prompt, so the run can never read or overwrite the real repo: `--disallowedTools Bash Read Edit Write Glob Grep WebFetch WebSearch Task TodoWrite NotebookEdit`.
- **Temperature — not pinnable.** The `claude` CLI exposes no temperature flag, so determinism cannot be forced. Compensate the way Phase 2 did: prefer **cross-project recurrence** (a delta seen in 2+ independent projects) over re-running one project, and reason only about **structural** deltas (role presence, pane count, release-flow shape, command specificity), treating exact role rosters and wording as noise. Where it matters, re-run 2–3× and note whether a delta is stable. If a true-determinism pass is ever needed, obtain an `ANTHROPIC_API_KEY` and switch to a raw-API renderer with `temperature 0`.

## The per-project input layout

Each project's prompt input is assembled from three files, concatenated in this order into `baseline-input.md` (and `baseline-input-v2.md` for a post-edit pass):

1. `rendered-prompt.md` — output of `render_config_gen_prompt` for the project dir (the prompt under test).
2. `capture-appendix.md` — the non-interactive override block: "you are being run to capture a baseline; tools are disabled; output a single rationale paragraph then one fenced `toml` block; write nothing." Reused verbatim across passes.
3. `project-snapshot.md` — the laid-out project (manifests, task runner, devbox scripts, CI, spec dir, agent launchers) standing in for step-1 discovery. Reused verbatim across passes; **regenerate it only if the target project itself changed.**

A post-edit pass changes **only** file 1 (re-render after editing the assets); files 2 and 3 are unchanged, so the v1↔v2 difference isolates the prompt edit.

## Authoring a new project's capture from scratch

The "Running it" steps below assume a project already has its `analysis/prd-116/<project>/` directory with `capture-appendix.md` and `project-snapshot.md` in place. **Adding a brand-new project is the main reason this procedure is documented**, so author those two inputs first — `rendered-prompt.md` is generated for you by the example in step 1, but the override block and the laid-out project are hand-authored once per project (and then reused verbatim across every pass).

```bash
export PATH="$PWD/.devbox/nix/profile/default/bin:$PATH"
P=<project-label>                       # the directory name under analysis/prd-116/
DIR=/home/vfarcic/code/<project>        # the real project to analyze
A=analysis/prd-116/$P
mkdir -p "$A"
```

**Step A — author `project-snapshot.md` (the project laid out).** Because the capture runs the model **single-shot with filesystem tools disabled**, the model cannot explore the repo itself — `project-snapshot.md` *is* its only view of the project, standing in for step 1's discovery. Author it by hand from the real repo so it carries every signal the prompt probes for. Mirror the structure of an existing snapshot (e.g. `analysis/prd-116/dot-ai/project-snapshot.md`); include, with project-specific content:

- A one-line **project-type** verdict and a `## Top-level entries` listing (annotated with what each entry implies, e.g. `devbox.json → init_command`, `prds/ → spec directory`).
- `## <reproducible-env manifest> (verbatim)` — paste the real `devbox.json` / `flake.nix` / etc., since it drives `init_command` and the agent launchers.
- `## CLAUDE.md (key sections)` — the test-mandate, release discipline, and any TDD/output-redirect conventions (these drive the `tester`/`coder` template specificity and the release gate).
- `## Task / build commands` — the real task-runner recipes or `package.json` scripts (abridged).
- `## agent launchers / skills` — any `devbox run agent-*`/`npm run agent-*`/`task agent-*` scripts and the `.claude/skills` (or opencode) coordination commands the orchestrator could run itself.
- `## CI/CD workflows` and `## Spec directory` — what gets validated on PRs, and the path workers read specs from.

Keep it faithful to the repo at the moment you capture it; **regenerate it only if the target project itself later changes.** Re-derive the config's last-edit date and the prompt-feature dates for the date gate (see below) at capture time.

**Step B — create `capture-appendix.md` from the template.** This override block is the same for every project except the project-dir in its final header line. Write it verbatim, substituting `$DIR` in the `## PROJECT LAYOUT` header (this header is what `project-snapshot.md` is concatenated under in `baseline-input.md`):

````markdown

---

## BASELINE CAPTURE MODE (PRD #116 — overrides the interactive steps above)

You are being run **non-interactively** to capture a reproducible baseline. Apply these overrides to the instructions above:

- **Filesystem and shell tools are DISABLED.** Do NOT read files, do NOT run `which`, do NOT write any file. The project has already been discovered for you and is laid out under "PROJECT LAYOUT" below — treat it as the complete result of step 1's discovery. Assume every binary in `devbox.json`/the listed toolchain is installed and on PATH, so **skip step 2's `which` validation**.
- **Do NOT ask the user anything and do NOT wait for confirmation.** Skip step 5's negotiation/numbering and step 6's file write entirely.
- **Output format:** first a short (≤1 paragraph) rationale of the project-specific signals you used, then the COMPLETE proposed `.dot-agent-deck.toml` (modes, plus an orchestration if one applies) in a **single fenced ```toml code block**. Output nothing after the code block.

## PROJECT LAYOUT (result of discovery for /home/vfarcic/code/<project>)
````

The block is intentionally short and project-independent; only the trailing `## PROJECT LAYOUT (result of discovery for …)` line names the project. After authoring both files, run the steps in the next section.

## Running it (one project, end to end)

```bash
# 0. Build the toolchain into PATH (rustup shims are broken here).
export PATH="$PWD/.devbox/nix/profile/default/bin:$PATH"

P=dot-ai            # project label under analysis/prd-116/
DIR=/home/vfarcic/code/dot-ai   # the project to analyze
A=analysis/prd-116/$P

# 1. Render the exact prompt the deck would send (no {dir}/{roles} left).
cargo run --quiet --example render_config_gen_prompt -- "$DIR" > "$A/rendered-prompt.md"

# 2. Assemble the model input (rendered prompt + capture override + project snapshot).
cat "$A/rendered-prompt.md" "$A/capture-appendix.md" "$A/project-snapshot.md" > "$A/baseline-input.md"

# 3. Generate the baseline single-shot, tools disabled, model pinned.
claude -p --model claude-haiku-4-5 \
  --disallowedTools Bash Read Edit Write Glob Grep WebFetch WebSearch Task TodoWrite NotebookEdit \
  < "$A/baseline-input.md" > "$A/baseline-raw-output.md"

# 4. Extract the FIRST fenced ```toml block into the baseline config.
#    Robust against a stray second ```toml block in the model output: capture
#    only the first block and `exit` the instant it closes, so nothing after the
#    first closing fence can ever be appended to baseline.toml.
awk '
  /^```toml/ && !started { started=1; infile=1; next }
  infile && /^```/       { exit }
  infile                 { print }
' "$A/baseline-raw-output.md" > "$A/baseline.toml"

# 5. Diff the baseline against the user-improved config, by region.
cargo run --quiet --example diff_config -- "$A/baseline.toml" "$DIR/.dot-agent-deck.toml" > "$A/diff.md"
```

For a **post-edit validation pass**, repeat with the `-v2` suffixes (`rendered-prompt-v2.md`, `baseline-input-v2.md`, `baseline-v2.toml`, `diff-v2.md`) after editing `assets/config_gen_prompt.md` / `assets/roles.toml` and rebuilding — only step 1 needs re-rendering; steps 2–3 reuse the unchanged `capture-appendix.md` and `project-snapshot.md`.

### Diff-tool pairing caveat

`diff_config` matches modes and orchestrations **by name**, and the generator picks those names nondeterministically (the prompt does not — and should not — dictate them). When a regenerated name differs from the user's, the tool reports the roles as disjoint (all B-only + U-only) instead of pairing them field-by-field, which *understates* the real overlap. To get a role-paired diff, copy the baseline and align only the cosmetic mode/orchestration `name = "…"` lines to the user's names before diffing — keep the authentic `baseline.toml` untouched and note the substitution at the top of the diff. (`name` lines for modes/orchestrations don't collide with role names, so a targeted string replace is safe.)

## The date gate (apply before cataloguing any delta)

A delta is only a user signal if the relevant prompt/role capability **existed when the user last edited the config**. Otherwise the divergence is time-drift — "generated by an older prompt and never regenerated" — not a preference. Before treating any region's delta as evidence, gate it:

- Find the config's last-edit date: `git -C <project> log -1 --format=%ci -- .dot-agent-deck.toml`.
- Find when the relevant prompt capability shipped in **this** repo: `git log` over `assets/config_gen_prompt.md` / `assets/roles.toml` (e.g. the prompt began *proposing orchestrations* on 2026-04-27, commit `3b83478`; the orchestration *engine* landed 2026-04-21).
- If the config predates the capability, **exclude** that region from the user-signal comparison and flag it (the pilot `dot-ai-infra` is excluded this way — its config predates orchestration-proposing, so its "no orchestration" is not a user choice). Otherwise the delta counts.

A pattern enters the catalogue only if it (a) clears the date gate **and** (b) appears in 2+ independent projects **or** is an obvious universal improvement. Single-project structural choices and nondeterminism artifacts are reported but not catalogued.

## Where the outputs go

- Per-project artifacts → `analysis/prd-116/<project>/` (captured prompts, raw outputs, extracted configs, region diffs; `-v2` variants for the post-edit pass).
- The aggregated pattern catalogue → `analysis/prd-116/catalogue.md`.
- The post-edit validation write-up → `analysis/prd-116/validation.md`.
- The project inventory / triage → `analysis/prd-116/inventory.md`.

Everything under `analysis/` is outside the published docs tree, so it stays internal automatically — keep it that way (no `site/sidebars.js` entry).
