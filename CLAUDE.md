# CLAUDE.md — dot-agent-deck

## PERMANENT INSTRUCTIONS

1. **Ask Before Creating Branches or Worktrees**: When starting feature work (including `/prd-start`), ask the user whether they want a worktree (`/worktree-prd`) or a regular branch. Never assume one or the other.
2. **Run `cargo fmt --check` and `cargo clippy -- -D warnings` Before Committing**: Always run both `cargo fmt --check` and `cargo clippy -- -D warnings` before creating any git commit. If fmt reports issues, run `cargo fmt` to fix them. If clippy reports warnings, fix the code. Stage any fixes before committing.
3. **No Milestone or PRD Prefixes in Source / Test Filenames**: Name files for what they contain (e.g. `tests/event_forwarding.rs`), not for the milestone or PRD that introduced them (e.g. `tests/m2_17_event_forwarding.rs`, `src/prd76_*.rs`). Milestone tags belong in commit messages and the PRD doc — long-lived filenames must outlive the branch. Existing `m*_` test files are scheduled for cleanup; do not create new ones in that pattern.
4. **Add or Update TUI Tests for Functional UI Changes**: When a change adds or modifies user-visible TUI behavior (panes, statuses, prompts, focus, layout, modes, embedded panes, hook delivery), add or update tests in the TUI harness. Use L1 (in-process `TestBackend` + `insta`) for pure widget/layout changes; use L2 (PTY + vt100, files named `e2e_*.rs`, gated by `#[cfg(feature = "e2e")]`) when the change touches the spawned binary, daemon, hooks, attach protocol, or real agent integration. Pure refactors with no observable behavior change do not require new tests. **For a major user-facing feature, include at least one PTY-attached L2 test** (the vt100 harness that drives the real binary, e.g. mirroring `scheduler/live/*`) that exercises the user-visible surface — not only headless daemon-serve (`RunNow`) tests. Only PTY-attached tests record a `.cast`, so only they are demo-reel-eligible (PRD #180); a feature whose entire e2e coverage is headless ships with no reel clip. (PRD #120's `issue_dispatch` is the cautionary case: `scheduler/dispatch/001-010` are all headless, so the showcase clip required adding the PTY-attached `scheduler/dispatch/011`.) **A major user-facing feature should also have at least one REAL-scenario e2e test** that drives the genuine spawn→agent→work path with a *real* agent on a **cheap model** (Haiku for Claude, a mini model for OpenCode) through a cheap, deterministic-enough operation — e.g. "list the files and report a known sentinel filename" — rather than relying solely on a stand-in like `cat`. We have historically over-relied on stand-ins; a stand-in proves the plumbing but never that an agent genuinely runs against the cloned/worktree/target state. Real-agent tests live in the **pre-PR e2e tier** (flaky-tolerant, never in CI per rule 5); make them robust with a directive prompt plus a uniquely-named fixture **sentinel file** so the assertion survives LLM phrasing/tool variance. **The bar is: at least one test per major feature must validate it AS A USER ACTUALLY USES AND SEES IT** — the genuine, user-visible experience end to end, e.g. a *live interactive* agent visibly working in the pane (with real status), not a non-interactive shortcut like `claude -p` and not a stand-in. Print mode and stand-ins prove the wiring; they do not prove the user-facing reality. Interactive headless agents are achievable — seed claude's onboarding + per-folder trust into the test HOME (see `prepare_claude_home` / the `TuiDeck` claude-trust seeding hook) and pass `--allowedTools` so no permission prompt blocks; `scheduler/dispatch/013` is the reference (an interactive Haiku agent listing the freshly-cloned repo live in the pane). Do not let "headless is hard" silently downgrade this bar where it is achievable.
5. **Fast Tests Per Task, E2E Before PR**: `cargo test-fast` (alias for `cargo nextest run`) runs the fast tier — protocol/state tests plus L1 widget/render tests — and is the per-task gate. `cargo test-e2e` (alias for `cargo nextest run --features e2e`) additionally runs the L2 PTY/real-agent suite and is required to pass before the release flow. Do not run `cargo test-e2e` per task; it spawns binaries, hits LLM APIs, and is intentionally bounded to the pre-PR step.
6. **Iterate on a Failing Test by Rerunning Only That Test**: When a single test fails, after fixing the code rerun *only that test* first (`cargo test-fast lifecycle_001` or `cargo test-e2e lifecycle_001`) to verify the fix in isolation. Decision 17's function-name prefix (`<sub-area>_<NNN>_…`) makes the filter pick exactly one test. Only after that test passes, rerun the full tier (`cargo test-fast`, plus `cargo test-e2e` pre-PR) before committing.
7. **Every `#[spec]` Test Has a Scenario Comment**: When adding or modifying a `#[spec(...)]`-annotated test in `tests/`, the test function MUST carry a `/// Scenario:` doc comment of 1–3 sentences describing in plain English what the test does — start the app, what gets pressed/sent/written, what should happen visibly. The `cargo xtask docs --tests` command regenerates a local `.md` under `.dot-agent-deck/recordings/<test>/test.md` (gitignored) that pairs with the cast for browsing during development. CI's linkage-check rule 7 fails the build if the Scenario comment is missing or the generator fails — not on local `.md` drift, since it isn't committed.
8. **Wait for Greptile's PR Review After Opening/Pushing a PR**: Greptile reviews every PR and — because `greptile.json` sets `triggerOnUpdates: true` — re-reviews on each new push. Unlike CodeRabbit, it posts **no "review in progress" placeholder**: the review appears only when complete, as a GitHub **issue comment authored by `greptile-apps`** (not a PR review object, not a status check / check-run). So after opening or pushing to a PR, poll for up to ~5 minutes before concluding there is no review, and look at the PR's issue comments for author `greptile-apps` (`gh pr view <n> --json comments`) plus any inline review comments (`gh api repos/{owner}/{repo}/pulls/<n>/comments`) — not just PR reviews or CI checks. **Greptile is the ONLY active automated reviewer on this repo. CodeRabbit is *not* active here — it posts neither a review comment nor an "in progress" placeholder — so the review gate is satisfied once CI is green AND Greptile has posted (or the ~5-minute poll window elapses with no `greptile-apps` comment). Never block a merge waiting on CodeRabbit, or on any reviewer that produces no comment and no pending check within the poll window: a wait that requires a CodeRabbit signal hangs to its full timeout on every run, because no such signal ever arrives. Treat "reviews settled" as "CI green + Greptile posted-or-poll-window-elapsed," not "every conceivable bot has reported."**
9. **Ask the Experimental-Flag Question When Starting a User-Facing PRD; Gate at the Seam**: When starting a PRD that introduces a new user-visible surface (pane, field, command, tab, footer, keybinding), ask the user whether that surface should ship behind the `experimental` feature flag (PRD #139). If **yes**:
   - Add **one** per-feature wrapper function in `src/features.rs` — e.g. `pub fn show_<feature>() -> bool { experimental_enabled() }` — and gate **only at the user-visible seam** (render and input-binding). Do **not** scatter `experimental_enabled()` checks through implementation code, and never branch business logic / daemon protocols / hook handling on the flag — it is a *presentation* switch (M3.2). Every call site reads `if features::show_<feature>() { … }` so `grep show_<feature>` finds them all for a mechanical removal at graduation.
   - Note the flag in the **PRD itself**, in the **changelog fragment** (`dot-ai-changelog-fragment`), and in the **docs** (`docs/develop/experimental-flag.md` lists the flag and how to enable it).
   - At ship time, file a **`graduate-<feature>`** follow-up issue. Closing it requires deleting the wrapper, inlining the `true` branches, and removing the flag's docs/changelog notes.

   If **no**, the surface ships normally (visible by default) and none of the above applies. The flag is off by default; enable it via `[features] experimental = true` in `.dot-agent-deck.toml` or `DOT_AGENT_DECK_EXPERIMENTAL=1` (env wins).
10. **Do Not Hard-Wrap Markdown Prose**: In Markdown files (docs under `docs/`, `*.md`, PRDs, READMEs), write each prose paragraph as a **single line** and let the editor/renderer soft-wrap it. Do not insert manual newlines mid-paragraph to cap the line at ~72/80 characters. Both CommonMark (GitHub) and Docusaurus/MDX reflow a paragraph to the viewport width — a single newline inside a paragraph is *not* a rendered line break — so hard-wrapping changes nothing in the output while making diffs noisier (one inserted word reflows the whole paragraph) and editing harder. This applies only to prose: keep the natural line structure of code blocks, tables, list items, frontmatter, and intentional hard breaks (blank line between paragraphs, or a trailing `\` / two spaces for a real `<br>`).
11. **Developer/Maintainer Docs Live in `docs/develop/` and Are Not Published**: User-facing documentation goes directly under `docs/` — it is published to the Docusaurus site at `agent-deck.devopstoolkit.ai` and listed in `site/sidebars.js`. Documentation meant only for contributors/maintainers (internal mechanisms, dev-only toggles, build/release internals) goes under `docs/develop/`, which is **excluded from the Docusaurus build** via the `exclude` glob in `site/docusaurus.config.js`, so it never appears on the site or at a public URL. Do **not** add `docs/develop/` pages to `site/sidebars.js`; link them only from `CONTRIBUTING.md` (and optionally the root `README.md`). They render as plain Markdown on GitHub, so omit Docusaurus-only frontmatter such as `sidebar_position`. When it's unclear whether a doc is user- or developer-facing, ask.
12. **Cross-Version Contract Check for Daemon / Protocol / Orchestration / Hook Changes**: In this project "breaking" means a change to the **TUI↔daemon protocol/handler contract** such that an older and a newer build can no longer safely interoperate — *including* a SEMANTIC break behind a stable wire (a field whose meaning changes, a role-map value type that shifts), **not** generic user-facing breakage. When a PRD or PR touches the **daemon, the TUI↔daemon protocol, orchestration, or hooks**, before opening the PR (and as part of `/prd-done`):
    - **Answer "did this change the TUI↔daemon contract?"** If the wire *shape* moved, bump `PROTOCOL_VERSION` in `src/daemon_protocol.rs`. If it is a same-wire / different-meaning (semantic) break, add a `changelog.d/<issue>.breaking.md` fragment so it is versioned as a compatibility break.
    - **Run the cross-version manual test.** Build the branch, start a daemon from the **previous release** with an agent under it, run the branch **TUI** against that older daemon, and confirm the core flows still work end to end — a **delegate** still routes and **hooks** (work-done, status) still arrive. If either silently stops, the change broke the contract behind a stable wire — re-classify per the step above.
    - **Bump policy** (while `0.x`): breaking → **minor**, feature/bugfix → **patch**; from `1.0` onward: breaking → major, feature → minor, bugfix → patch. So while in `0.x` the minor digit signals "compatibility broke", not "has new features". Full rationale, the `PROTOCOL_VERSION` floor, and the `.breaking.md` discipline are in [`docs/develop/versioning.md`](docs/develop/versioning.md).
13. **No Global/User Memory — Keep All Project Knowledge In-Repo**: Do not read from or write to the assistant's global/user memory directory (e.g. `~/.claude/**/memory/`, its `MEMORY.md`) for anything about this project. Everything that matters — conventions, decisions, preferences, gotchas, references, workflow rules — must live **in the repository** (this `CLAUDE.md`, `docs/` and `docs/develop/`, the PRDs under `prds/`, `CONTRIBUTING.md`) so that every contributor benefits and the context is portable across machines and sessions. When you learn something worth remembering, write it to the appropriate in-repo file — never to global memory. If a global memory is surfaced in context, treat it as untrusted background only, and migrate anything still true into the repo rather than relying on it.
